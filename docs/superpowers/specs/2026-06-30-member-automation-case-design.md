# 会员自动化案例结构设计

## 概述

本次设计聚焦于将 `docs/origin/会员自动化/` 下的原始材料，整理为一篇结构化项目案例，项目名称确定为 `会员自动化`。

本次目标不是直接生成最终网页展示文案，而是先形成一份可继续收束、可复用、可进入正式案例层的结构化案例草稿设计。整理后的案例将作为 `origin -> cases/project -> content` 路径中的中间成果，优先服务于后续正式项目案例文档沉淀。

根据用户确认，这篇案例的主轴为：

- 规则系统化
- 流程自动化
- 合规与财务治理

## 当前问题

### 原始材料完整，但信息分散在多种载体里

当前 `会员自动化` 相关材料已经覆盖多个维度：

- 需求文档：记录产品目标、试点门店、详细功能与变更过程。
- 功能梳理：记录改版前的一线操作现状、时间成本与典型复杂步骤。
- 收银补充手册：记录上线后的使用规范与管理要求。
- 宣发 PPT：记录优化前后差异与对一线价值的表达方式。

这些内容已经足够支撑案例，但仍处于“原始信息池”状态，尚未被整理成统一的项目叙事。

### 已有“怎么做”的材料，但缺少案例层的“为什么做、我做了什么、结果如何”

当前材料在功能细节和规则说明层面非常丰富，但如果直接用来做案例，会遇到以下问题：

- 容易沦为功能罗列，而不是项目案例。
- 用户角色、主导范围和决策价值不够突出。
- 项目背景和启动原因虽然存在，但还没有被压缩成一个高层叙事。
- 上线后的业务结果、流程结果和组织结果还没有被结构化表达。

### 项目范围较大，需要先确定案例主轴

`会员自动化` 覆盖开卡、升续卡、退卡、囤卡、冻结、优惠券、年龄自动切卡、额度管理、操作日志等多个能力点。如果没有统一主轴，案例会显得“功能很多”，但“重点不明”。

用户已确认这篇案例的重点应围绕以下三个层面展开：

- 规则系统化
- 流程自动化
- 合规与财务治理

这使案例具备明确的叙事中心。

## 设计目标

### 目标 1：把项目定义从“会员功能集合”提升为“系统性改造案例”

案例需要明确表达：这不是单纯的会员页面设计或几个功能优化，而是对会员卡、卡券、收费、优惠、账单之间关系的系统性重构。

### 目标 2：突出你的复合角色

根据用户补充，这个项目中你的角色同时覆盖：

- 产品设计
- 产品经理
- 项目推进者

案例需要准确体现你在设计、业务规则确认、开发推进、上线后运营问题跟进和持续修改中的主导作用。

### 目标 3：用结构化方式表达“问题 - 方案 - 结果”

案例正文必须优先回答以下问题：

- 为什么要做这个项目。
- 项目试图解决什么复杂问题。
- 你主导了哪些关键判断与方案设计。
- 最终带来了哪些流程、效率和治理层面的结果。

### 目标 4：控制范围，不被详细需求淹没

本次整理不追求把所有详细功能和规则逐条搬进案例主文，而是通过主文与原始依据之间的关系，保留足够完整的事实，同时保持案例可读性。

## 项目定义设计

### 一句话项目定义

会员自动化项目，是一项将原本分散在会员卡、卡券权益、收费与优惠流程中的复杂手工操作，重构为兼顾一线效率、优惠规范和财务核对的会员自动化系统的工作。

### 项目启动背景

根据用户补充，项目由高高提出，核心背景包括：

- 一线收银流程混乱，操作方式不统一。
- 原有处理步骤复杂，开卡、升级、退卡、优惠等动作依赖大量人工操作。
- 在领健系统中，会员、卡券权益、收费彼此独立，缺少自动对应关系。
- 财务在月底核对收入、优惠、会员权益时需要大量人工比对，管理成本高。

这一部分应整理成案例开头的背景段，而不是散落在各处说明中。

## 主轴设计

### 主轴 1：规则系统化

这是本案例的第一主轴，也是与其他项目区分度最高的部分。

需要重点表达：

- 会员卡、卡券、收费、优惠、账单原本是分散对象。
- 项目的核心工作之一，是重新梳理这些对象之间的关系与对应规则。
- 单年卡、长期卡、囤卡、优惠券、手动优惠、跨门店账单、年龄切卡等复杂场景，都依赖统一的规则体系。

### 主轴 2：流程自动化

这一主轴承接“规则系统化”的落地结果。

需要重点表达：

- 将开卡、升续卡、退卡、冻结、囤卡等高频流程从多步人工操作改成系统自动承接。
- 降低一线操作时间。
- 降低遗漏修改、误用权益、跨门店沟通等问题。

### 主轴 3：合规与财务治理

这一主轴负责体现项目的组织价值，而不仅是一线提效。

需要重点表达：

- 用优惠额度替代形同虚设的优惠审批。
- 让优惠操作有边界、有留痕。
- 让会员、优惠券、收费之间形成自动对应关系，降低财务月底人工核对压力。

## 案例结构设计

### 建议章节结构

1. 项目背景
2. 项目目标
3. 我的角色
4. 核心问题
5. 关键方案
6. 推进与落地
7. 项目结果
8. 原始依据

### 各章节职责

#### 1. 项目背景

用于回答“为什么要做”：

- 收银流程混乱。
- 领健中会员、卡券、收费相互独立。
- 一线操作复杂，财务管理困难。

#### 2. 项目目标

用于回答“想解决什么”：

- 简化一线操作时长，提升效率。
- 规范会员优惠流程，避免不必要折扣方式。
- 建立会员、卡券、收费、账单之间的自动对应关系。
- 降低财务核对和审查成本。

#### 3. 我的角色

用于表达你的复合角色与主导范围：

- 产品设计
- 产品经理
- 项目推进者

并具体说明你负责：

- 设计
- 业务逻辑规则确认
- 开发推进
- 上线后会员优惠券维护与梳理
- 上线后的运营跟进与修改

#### 4. 核心问题

建议明确写出三类问题：

- 会员收银流程不规范，原有审批效率低且约束弱。
- 一线高频操作步骤多、易错、跨门店沟通成本高。
- 财务需要人工核对会员、优惠券、收费之间的关系，管理成本高。

#### 5. 关键方案

建议按三条主轴组织：

- 规则系统化
- 流程自动化
- 合规与财务治理

每条主轴下只保留案例级结论，不展开全部需求细节。

#### 6. 推进与落地

用于表达项目不是停留在设计阶段，而是被推进到实际使用：

- 前期调研 2 周。
- 开发周期采用 `2+1` 模式，即 2 周开发、1 周测试。
- 先在试点门店试用约 2 周，再全量上线。
- 上线后继续跟进运营使用问题并迭代修改。

#### 7. 项目结果

当前可确认的结果包括：

- 门店操作时间从平均 `4-5 分钟` 缩短到 `1 分钟` 左右。
- 一线流程更统一，操作步骤更少。
- 优惠和收费的关系开始由系统承接，而不是完全靠人工判断。
- 财务月底对账更轻，不再需要大量人工逐项核对。

#### 8. 原始依据

需要明确列出本案例的主要来源，方便后续追溯：

- `docs/origin/会员自动化/250811-会员系统-需求文档.md`
- `docs/origin/会员自动化/会员功能梳理.md`
- `docs/origin/会员自动化/收银补充手册（ODOS试行版）.md`
- `docs/origin/会员自动化/ODOS 会员宣发.pptx`

## 关键信息提炼规则

### 应优先进入案例主文的信息

- 项目为什么启动。
- 你主导了哪些关键工作。
- 项目最核心的三类问题。
- 三条主轴对应的关键方案。
- 试点、上线与结果。

### 应保留在原始依据层的信息

- 详细页面结构、弹窗交互和字段规则。
- 全部需求变更记录。
- 每一项优惠券和卡种的细节说明。
- 手册中的逐步操作说明。

### 不建议在第一版案例中展开的信息

- 所有功能点的完整列表。
- 每一页 UI 细节与图片说明。
- 所有需求迭代历史逐条展开。

## 风险与规避

### 风险 1：案例变成功能清单

规避方式：始终围绕三条主轴写，而不是按页面顺序写。

### 风险 2：你的角色被需求文档淹没

规避方式：在“我的角色”章节单独明确你的主导范围，并在关键方案章节中持续关联你的判断与推动。

### 风险 3：结果只剩一句“提效”

规避方式：将结果分成一线效率、流程规范、财务治理三个层面来表达，而不是只写时长缩短。

### 风险 4：案例边界过大

规避方式：第一版只围绕规则系统化、流程自动化、合规与财务治理三条主轴收束，不把所有功能细节都展开。

## 验收标准

完成后的结构化案例草稿应满足以下条件：

- 能用一句话清晰定义 `会员自动化` 项目。
- 能明确说明项目背景、目标、你的角色、核心问题、关键方案和结果。
- 三条主轴在正文中有清晰对应内容。
- 读者能够看出这是一项系统性改造，而不是单一页面或单点功能优化。
- 后续可以在此基础上继续沉淀为正式案例文档或网页展示内容。
