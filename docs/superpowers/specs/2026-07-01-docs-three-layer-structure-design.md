# 三层文档结构设计

## 概述

本次设计用于明确 `docs/` 目录的长期结构，确保项目材料从原始文件、事实整理到网站表达之间形成稳定、清晰、可维护的三层路径。

本次设计的核心目标不是增加更多文档，而是固定文档流转方式，避免后续在归档、整理、网站提炼之间反复混淆目录职责。

## 已确认的文档逻辑

根据当前对话，已确认采用以下三层结构：

- `docs/origin/`：存放原始材料，作为归档与追溯依据。
- `docs/project/`：存放整理后的项目事实文档，用于面试复盘、梳理思路与后续内容提炼。
- `docs/site/`：存放面向网站的提炼规则、结构说明与页面内容映射。

同时确认以下边界：

- `docs/cases/` 不再保留，后续并入 `docs/project/`。
- `docs/project/` 中的项目文档统一采用“事实整理版”口径，而不是网站展示稿。
- 文档规则采用“每层各自 README”的方式固定，避免只写一份总规则后续难以维护。

## 设计目标

### 目标 1：固定三层职责

每一层只承担一个明确职责：

- `origin` 负责保留原始文件与过程材料。
- `project` 负责沉淀经过整理的事实版项目文档。
- `site` 负责提炼网站表达，不直接承担事实归档。

### 目标 2：统一项目文档入口

所有整理后的项目文档统一放在 `docs/project/`，避免 `cases`、`project`、`site` 之间重复承载项目内容。

### 目标 3：将规则写入文档

通过在 `origin/`、`project/`、`site/` 各自的 `README.md` 中写明规则，使后续整理时可以直接参照，而不是依赖口头约定或对话记忆。

## 目标目录结构

```text
docs/
├── origin/
│   ├── README.md
│   └── ...原始资料、表格、PPT、过程草稿
├── project/
│   ├── README.md
│   ├── 会员自动化.md
│   ├── 正畸筛查与状态管理.md
│   └── ...其他整理后的项目事实文档
├── site/
│   ├── README.md
│   ├── content-map.md
│   └── structure.md
├── profile/
│   ├── README.md
│   └── ...个人背景与定位文档
└── superpowers/
    ├── plans/
    └── specs/
```

## 各层职责定义

### `docs/origin/`

定位：原始材料区。

这里保留的内容包括：

- 原始需求文档
- 过程草稿
- 表格
- PPT
- PDF
- 会议整理稿
- 尚未经过结构化梳理的补充材料

这一层的规则是：

- 以归档为主，不承担对外表达职责。
- 可以作为后续整理依据，但不直接作为网站文案来源。
- 不要求统一表达风格，只要求文件来源可追溯。

### `docs/project/`

定位：项目事实整理区。

这里保留的内容包括：

- 已经整理完成的项目事实文档
- 用于面试复盘的项目说明
- 供后续网站内容提炼的基础材料

这一层的规则是：

- 每个项目只保留一份主文档。
- 统一采用“事实整理版”口径。
- 重点写清：项目背景、问题、我的角色、关键动作、结果、原始依据。
- 不急于使用作品集语言，不直接追求网页展示效果。
- 当项目仍有未决细节时，可以保留边界说明，但应区分“已确认事实”和“当时仍在确认”的内容。

### `docs/site/`

定位：网站提炼区。

这里保留的内容包括：

- 网站结构说明
- 页面内容映射
- 提炼规则
- 页面表达边界

这一层的规则是：

- 网站表达优先从 `docs/project/` 和 `docs/profile/` 中提炼。
- 不直接从 `docs/origin/` 抽内容进入页面。
- 不承担原始事实归档功能。
- 关注“如何呈现”，而不是重新定义“事实是什么”。

## 文件迁移与调整范围

### 需要处理的现有目录

- `docs/cases/`：后续并入 `docs/project/`。
- `docs/project/`：补充 README 规则，作为整理后项目文档唯一入口。
- `docs/origin/`：新增 README，明确其归档职责。
- `docs/site/`：更新 README，明确其仅承担网站提炼职责。

### 需要处理的现有文件

- `docs/cases/会员自动化.md`：迁移到 `docs/project/会员自动化.md`
- `docs/project/README.md`：更新为“事实整理版项目文档”的规则说明
- `docs/site/README.md`：更新为“三层结构中的网站提炼层”说明
- `docs/site/content-map.md`：如有必要，同步修正文案来源描述，使其与 `project` 的唯一入口保持一致

## README 规则设计

### `docs/origin/README.md`

需要明确：

- 这里放什么
- 这里不放什么
- 与 `project` 的关系
- 为什么不能直接从这里写网站内容

### `docs/project/README.md`

需要明确：

- 这里是整理后项目文档的唯一入口
- 文档统一为“事实整理版”
- 推荐的项目文档结构
- 与 `origin`、`site` 的上下游关系

### `docs/site/README.md`

需要明确：

- 这里负责网站内容提炼与映射
- 内容来源优先是 `project` 和 `profile`
- 不直接从 `origin` 提取网页内容

## 项目文档推荐结构

`docs/project/` 中的单个项目文档，推荐统一使用以下主结构：

1. 项目背景
2. 核心问题
3. 项目目标
4. 我的角色
5. 关键动作
6. 项目结果
7. 原始依据

如项目复杂度较高，可以在主结构后增加附录，用于保留：

- 复杂规则摘要
- 当时的边界信息
- 未决问题
- 补充事实材料

但主结构仍应优先保持一致，便于后续提炼与对比。

## 文档流转规则

后续新增项目时，默认遵循以下流程：

1. 先将原始材料归档到 `docs/origin/`
2. 再整理为 `docs/project/` 中的事实版项目文档
3. 最后根据需要提炼到 `docs/site/` 对应页面说明中

这条流转路径应写入各层 README，作为长期规则保留。

## 非目标

本次设计不做以下事项：

- 不直接重写所有现有项目文档内容
- 不直接改写网站页面代码
- 不新增独立的“interview”或“cases”中间层
- 不把 `docs/site/` 变成项目事实归档区

## 预期结果

完成后，文档结构应达到以下状态：

- 三层结构职责清晰，后续不再混用。
- `docs/project/` 成为整理后项目文档的唯一入口。
- `docs/cases/` 退出后续使用路径。
- 每层都有自己的 README 规则说明。
- 后续补充项目材料时，可以按固定流程直接落位，而不会依赖临时判断。
