# 定位与内容整理原则设计

## 概述

本次调整不直接扩写个人经历或项目案例，而是先将当前已确认的求职定位与内容整理原则写入文档体系，作为后续补充 `docs/profile/` 与 `docs/site/` 的依据。

本次设计服务于两个目标：

- 固定当前求职方向，避免后续在“产品经理”与“设计师背景”之间来回摇摆。
- 固定内容整理顺序，避免在事实尚未梳理清楚时过早进行包装式表达。

## 已确认的信息

根据当前对话，已确认以下内容：

- 当前目标岗位为：`产品经理 + 设计复合型岗位`
- 首页第一屏的主表达为：`产品经理 / 设计复合型人才`
- 行业标签 `儿童口腔医疗数字化` 不在首页第一屏强打，而是在后续区块逐步展开
- 文案风格偏：专业表达
- 当前最希望被招聘方感知到的核心能力为：产品方案设计能力
- 当前更适合采用的整理路径为：
  `先整理事实 -> 再提炼能力和方法 -> 最后生成网站表达`

## 设计目标

### 目标 1：在 `profile` 中写清定位

`docs/profile/` 需要承载个人定位本身，而不是只做背景资料堆积。此次补充应明确：

- 主身份是产品经理，而不是“偏产品的设计师”
- 设计背景是差异化优势，而不是主身份
- 当前内容提炼以事实为基础，而不是先包装人设

### 目标 2：在 `site` 中写清表达原则

`docs/site/` 需要承载网站页面层面的表达策略。此次补充应明确：

- 首页第一屏先建立通用求职身份
- 行业标签放到后续模块展开
- 页面内容优先从已整理的事实材料中提炼，不直接跳过整理过程反向包装

### 目标 3：保持现有文档结构稳定

本次仅在现有文件中增补说明，不新增新的定位子目录或额外文档层级，以避免文档结构继续膨胀。

## 文件修改范围

### 修改：`docs/profile/README.md`

补充“当前整理原则”说明，明确：

- 当前求职方向
- 当前文档整理顺序
- `profile` 在整个文档体系中的职责

### 修改：`docs/profile/overview.md`

补充“当前求职定位”与“当前内容提炼原则”两部分，明确：

- 当前求职定位为 `产品经理 / 设计复合型人才`
- 表达风格偏专业、克制
- 重点强调产品方案设计能力
- 当前阶段不急于抽象包装，而是从真实经历和项目事实中提炼内容

### 修改：`docs/site/content-map.md`

补充页面表达原则，明确：

- `Hero` 先传达通用身份
- 行业标签放在后续模块中逐步展开
- 网站内容优先从 `docs/project/` 与 `docs/profile/` 中整理后的内容提炼

## 文案原则

### 原则 1：产品身份优先

文案中应将“产品经理”作为主身份，设计背景作为能力增强项，而不是并列两个松散标签。

### 原则 2：专业表达优先

语言应偏专业、克制、清晰，减少空泛表述，例如：

- 少用“热爱”“擅长很多”“全面发展”
- 多用“如何理解问题”“如何形成方案”“如何推动落地”

### 原则 3：事实优先于包装

当前阶段文档补充应以“整理原则”和“定位约束”为主，不直接大篇幅改写个人故事。真正的网站文案，应在后续项目与经历整理完成后再统一生成。

## 预期结果

完成后，文档体系应达到以下状态：

- `docs/profile/` 已明确当前求职方向与内容提炼逻辑
- `docs/site/` 已明确首页与后续模块的表达边界
- 后续整理项目案例时，可以直接沿用“先事实、再提炼、后表达”的路径

## 非目标

本次不做以下事项：

- 不新增单独的“定位说明”目录
- 不直接扩写完整首页文案
- 不直接重写 `About`、`Experience`、`Portfolio` 页面内容
- 不替代后续项目案例整理工作
